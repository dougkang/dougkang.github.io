Why I Switched to Jekyll

  • Blog
  • Projects

and why I ditched Wordpress

11 Apr 2013

I’ve been using Wordpress for a good 6 or 7 years now and I got pretty good at it, making a couple of plugins and full-fledged themes and such, but when it came down to it, Wordpress just wasn’t cutting it for me, for one main reason: it’s too darn heavyweight. For a “simplified” blogging interface (and I use that term lightly), I had to stand up a db and execute php, all for a site that barely changes weekly? I don’t need comments, I don’t need to change themes, I don’t need roles, I just need blogging

So along comes Jekyll. I found out about it through a company that was working next to us during my earlier startup days. You write in Markdown in whatever editor you like. You compile your blog like a program and it jekyll spits out the site in plain ol’ html which you can serve however you like. As a backend programmer, this makes sense to me. My site doesn’t change on even a daily basis - why does my blogging framework need to dynamically create the site more or less on every visit?

I laughed a little to myself though, because as I was just starting to learn how to program, I wrote a similar (albeit much more simplistic) thing in C. I called it Janga (it was a time when Xanga was all the rage). I had a directory with my posts, a header and a footer. I would run my Janga program and it would spit out an html sandwich of the header, all the posts, and the footer, which I would then manually upload to Geocities. Of course, being only like 12 at the time and just starting out, it was super janky, but it worked and I was happy.

So I’ve been tinkering around with Jekyll as mentioned before and I think I got a pretty good workflow going on, so I thought I might share it.

A little about my site:

  • powered by jekyll

  • using flat-ui for styling

  • three layouts

    • the default, which is the layout for viewing multiple posts or projects
    • the post, which is the layout for viewing single posts
    • the project, which is the layout for viewing single projects

Now the setup:

I have my blog as a git repository.

I have a remote git repository on my server that I have set up with a post-receive hook on it that I stole from here.

I have a script to add that I stole from here and modified to automatically add the post to git and open the resulting file in Sublime Text.

I have Sublime Text installed with the Markdown Preview plugin installed. Make sure you have the Live Reload plugin installed as well. The only weird this with this is that Markdown Preview completely barfs when it tries to read your YAML front matter. Oh well.

My workflow looks something like this: 

  $MY_WEBSITE_REPO/tools/add_script.rb
  # This opens up a Sublime Text window with the newly created post opened
  # I bring up Markdown Preview, which brings up a browser which renders the markdown on each save
  # When I'm done writing:
  git commit -m "My new post"
  git push toserver

Once I push, the post-receive hook kicks in, generates the site and moves it to the appropriate location if the compilation was successful. My site has been updated with my new post. Pretty sweet if you ask me!

I’ve seen how easy it is to use plugins but I haven’t used any yet. I’ll let you know how that turns out

The not so good

But, here are the not so cool things that I’ve found:

  • Adding images is not so fun. It’s not as braindead as it was in wordpress.
  • While the framework is pretty flexible, its still blog oriented, so it’s limited to blog-like websites.
  • Migrating my wordpress was a pain. I used exitwp, but there was still alot of manual work that I needed to do and all my image links are broken.

Since, I’ve streamlined my blogging process as much as I can, hopefully that will drive me to write more to this thing. I have a couple of things that I really want to show regarding fantasy basketball that I’ve been refining over the course of the last season.
